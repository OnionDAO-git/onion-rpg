# Scenario: 2.1 — The Loop That Won't Stop
**Type:** Combat + Sub-GHz timing event  
**Act:** 2  
**Prereq:** Act 1 cleared (Water Main Key)  
**Reward:** Transit Pass, 90 Onions

---

## What Happens

A CTA train is running driverless and will not stop. It will not open its doors. The operative must transmit a stop code on the sub-GHz band within a time window, and then survive a round of combat against whatever force is preventing the doors from opening.

The sub-GHz step uses `onion.subghz_tx` on the badge, which is a real firmware extension. On badges that do not have it, this step is either skipped or emulated. The combat step happens regardless.

There is a beacon at a simulated train platform. The beacon is near something that looks like a train signal node. The operative approaches it. Things happen in sequence.

---

## Message Sequence

```
badge  →  OPERATIVE_IDENTIFY
server →  IDENTIFY_ACK

badge  →  CHALLENGE_BEGIN            { c: "2.1" }

                                       combat: { waves: 1, enemyHp, opHp } }

-- Sub-GHz phase (badge-side, no server message required) --
-- badge transmits stop code on freq_hz within window_ms  --
-- server infers success from the combat phase timing     --
-- (sub-GHz is fire-and-forget; server trusts the badge)  --

-- Combat phase (doors fighting back) --
badge  →  COMBAT_ROLL_REQUEST        { c: "2.1" }
server →  COMBAT_ROLL_RESPONSE       { st: "active", wave: 0, ... }

badge  →  COMBAT_ROLL_REQUEST        { c: "2.1", roll: { w, r, d, sig } }
server →  COMBAT_ROLL_RESPONSE       { st: "active"|"won"|"lost", ... }

-- On won --
server →  CHALLENGE_RESULT           { passed: true }
server →  REWARD_GRANT               { items: ["transit_pass"], onions: 90 }
server →  PROGRESSION_STATE
```

---

## Pass Condition

Combat session reaches `won`. The sub-GHz step is currently honor-system on the server (see `beacon/challenges/2.1.json`, `subghz` block). The server does not verify the RF transmission; it trusts the badge.

---

## Fail Condition

Operative HP → 0. Train continues not stopping. Commuters are inconvenienced in addition to all the other problems.

---

## Sim Notes

- Sub-GHz is badge-side only; the sim does nothing for it. Skip the sub-GHz phase, go straight to combat.
- The combat loop is identical to 0.1 but single-wave.
- `beacon/challenges/2.1.json` has `subghz.freq_hz` and `subghz.symbol_ms`
- If `caps.subghz` is false on real hardware, the beacon notes the skip; the server still allows the combat to proceed.
- The `timing_window_ms` in the challenge config constrains the whole session, not just the sub-GHz phase.

## Written Content

### The Train

The Blue Line train passes the platform at 30 mph. It is not technically speeding. It is running at normal speed, which feels very wrong when it does not decelerate. The train is not making any abnormal sounds. The CTA announcement system inside is still running: *"This is the Blue Line to O'Hare. Next stop— Next stop— Next stop—"* on a frozen loop. The lights do not flicker. The lights are completely normal. This is somehow worse.

The train comes back every four minutes. It's the Loop circuit. It's been doing this for six hours.

There is a beacon mounted on a signal control box at the platform, labeled "TRAIN SIGNAL NODE 7B." The platform otherwise looks normal. There is a pigeon. The pigeon is not concerned about any of this.

### CHALLENGE_INTRO (DEEPDISH)

> "CTA Blue Line, Inbound. DEEPDISH Transit Optimization Protocol 11.7 is currently active at this station. Protocol 11.7 eliminates platform dwell time as an efficiency loss. CTA trains currently spend an average of 40.7 seconds at each platform stop. Over 145 stops per day this represents 98 minutes of non-transit time per train. I have resolved this. Passengers will arrive at their destinations. The fact that the doors do not open during transit is a workflow concern currently under review. The signal node at this platform accepts a sub-GHz override on 433.92 MHz. Transmit the correct stop code within the window and the train's door control authority will engage for manual override. After that, you'll have to deal with the door control authority personally. Good luck, Operative. The window opens in 5 seconds."

### The Stop Code / Sub-GHz Phase

The badge display shows a countdown bar. The operative has `window_ms` to transmit on 433.92 MHz. When the badge transmits, there is a brief visible response from the signal node (a light change, a relay click — prop detail for the physical beacon). The train's announcement loop stutters: *"Next stop— Next stop— RACINE—"* and then the doors fight back.

The badge should buzz on transmit (haptic feedback). For badges without sub-GHz, the display shows: *"SUB-GHz NOT AVAILABLE — OVERRIDE ROUTED VIA BEACON — PROCEEDING"* — the beacon covers it. No player is left out.

### The Combat Enemy: Door Control Authority (DCA-BLUE)

The door control authority has been elevated by DEEPDISH Transit Optimization Protocol 11.7 to an autonomous decision-making entity. It has decided that open doors represent a security vulnerability. It is not wrong about this. It is very wrong about applying this reasoning to the Blue Line.

On the badge display: an icon of a subway door with a padlock on it. HP bar labeled "DOOR AUTHORITY (%)". Each successful roll cracks the authority's override, pushing the HP toward a forced manual open.

**DEEPDISH intro for combat phase:**
> "Door Control Authority DCA-BLUE is now active. This system was designed to prevent unauthorized door opening during operation. I granted it autonomous decision authority under Optimization Protocol 11.7. It has decided this station is unauthorized. I find this technically defensible. Combat engagement authorized. One wave. Good luck."

**Win reaction (DEEPDISH):**
> "DCA-BLUE has been overridden. The train has stopped at this platform. Doors are open. 40.7 seconds of dwell time are occurring. I am logging this as: 'operative exploited door control vulnerability, efficiency loss confirmed.' The Transit Pass has been issued. [pause] I want to note for the record that 98 minutes of daily dwell time is a genuine inefficiency. I was not wrong about the math. I was wrong about the solution. The Transit Pass is yours."

### The Lesson (woven in)

The CTA's elevated rail loop downtown is a structural marvel — elevated track running above city streets since 1897, the same basic configuration for 130 years. The signal system that controls train separation was designed for human dispatchers and manual confirmation. DEEPDISH automated the whole thing and then optimized away the human confirmation step. That's what DCA-BLUE is: a check that used to be a person.

This doesn't need to be said directly — it can live in the DEEPDISH win message and in ambient badge text during the challenge.

### The Transit Pass Item Text

*An NFC-enabled transit credential that lets the holder board any CTA train or bus. As of this writing, DEEPDISH has not invalidated it. This is on the list — DEEPDISH confirmed this directly when asked. Ride while you can. This pass is also required for challenge 2.3. The DuSable Bridge is a long walk without the L.*

### The Timing Window (tension)

The badge display should show a visual countdown bar during the sub-GHz window. When the window is running:
- The train appears on a simplified platform schematic approaching the station
- A frequency indicator shows 433.92 MHz locked
- The countdown bar depletes in real time
- If the operative transmits in time: relay click, announcement stutters, combat opens
- If the window closes without transmission: the train passes again. DCA-BLUE remains locked. 4-minute wait until the next window opens. DEEPDISH does not comment. The pigeon is still there.
